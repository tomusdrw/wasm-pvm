# anan-as interpreter core-lowering miscompile — reproduction

Self-contained reproduction for **issue #261**: `wasm-pvm`'s always-on core
lowering miscompiles the anan-as PVM interpreter. The compiled interpreter
mis-executes certain **valid** inner PVM programs (returns an empty result),
while the interpreter *source* runs the same programs correctly.

This is **pre-existing on `main`** and not caused by any optimization pass.

## TL;DR

```bash
./experiments/interpreter-miscompile/reproduce.sh
```

Builds the interpreter from current source, runs the committed `trigger.jam`
three ways, and prints a verdict. Exit code **1** = bug reproduced, **0** =
the compiled interpreter ran it correctly (bug fixed).

Expected output today (bug present):

```text
A. wasm-pvm-compiled interpreter : WRONG - result=0x (empty)
B. native anan-as engine (direct): OK - result=0x11000000
C. interpreter source (native)   : OK - len=21 result=0x...11000000

REPRODUCED (#261): the inner program is valid (B and C agree),
but the wasm-pvm-compiled interpreter mis-runs it (A wrong).
```

## What it shows

`trigger.jam` is a small (~700 B), **valid** PVM program — a compiled
`as-tests-globals` that returns `17` (`0x11000000`) for input `00000000`. The
harness runs it three ways:

| Check | How | Expected | Actual today |
|---|---|---|---|
| **A** | through the wasm-pvm-compiled interpreter (`anan-as-compiler.jam`), via the PVM-in-PVM CLI path (`runCompilerJam`) | `17` | **empty** ❌ |
| **B** | directly on the native anan-as engine (`prepareProgram` + `runProgram`) | `17` | `17` ✅ |
| **C** | through the interpreter **source** `compiler.wasm` executed natively (Bun WebAssembly) | `17` | `17` ✅ |

B and C are two independent references confirming the inner program is valid.
Only A — wasm-pvm's PVM compilation of the interpreter — gets it wrong.
Therefore the bug is in wasm-pvm's code generation for the interpreter.

## Why a committed `trigger.jam`

The bug is **input-layout-dependent**: whether a given valid inner program
exposes it depends on the program's exact byte layout (the offsets / jump-table
values / block structure that the interpreter decodes and computes on). On
`main`, the standard test programs happen to avoid the triggering layout (which
is why layers 4–5 are green there). `trigger.jam` is a known-triggering layout,
committed so the reproduction is hermetic and doesn't depend on any
optimization branch or compiler flag.

It was originally produced by compiling `tests/fixtures/assembly/tests-globals.ts`
with the dead-code-elimination pass enabled (branch `perf/dce-absfold`), which
shifts byte layout into the triggering range. DCE is *correct* — it only
changes layout — but it exposes this latent bug, which is why the DCE +
absolute-address-folding optimizations are blocked on #261.

| `run-checks.ts` | Self-contained 3-way checker (`bun run-checks.ts <interp.jam> <trigger.jam>`). |
| `trigger.jam` | The committed triggering inner program (compiled `as-tests-globals`). |

`anan-as-compiler.jam` is generated by `reproduce.sh` and git-ignored.

## What's been ruled out (see #261)

- Not an optimization pass — a full `--no-*` sweep of the interpreter
  compilation fixes nothing → always-on core lowering.
- Not the inner program / not DCE — two native engines run it correctly; DCE
  only changes layout.
- Not anan-as — the interpreter source is correct natively.

## Notes for whoever fixes it

- Use this harness (or the `bun test layer4/ layer5/` suite) as the oracle. Do
  **not** trust a hand-rolled `node index.js run` loop as a quick check — it is
  flaky under memory pressure ("Run out of pages! Allocating.") and can report
  false PASS/FAIL. `run-checks.ts` mirrors the real test path.
- The mis-lowered construct was not isolated (the interpreter is ~22k PVM
  instructions; passing vs failing inner programs diverge in a data-dependent
  way, so trace-diffing is inconclusive). Suspected areas: varint/immediate
  decode of inner-program offsets, basic-block / skip-distance computation,
  block-gas accounting, unaligned multi-byte loads.
- To confirm a fix end-to-end: rebase `perf/dce-absfold` onto the fix and
  verify `cd tests && bun test layer4/ layer5/` is green.
